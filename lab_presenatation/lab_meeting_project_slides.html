motivation

Why study MAS development practice?

Multi-agent systems are increasingly used for complex workflows, but the tooling story is still immature. I want to understand how developers actually build them, where time goes, and what kinds of support would matter in practice.

ground tool ideas in observed workflows focus on pain points, not idealized pipelines

working definitions

for this project

AI agent

An LLM-guided component with a role, prompt, context, and some decision or action responsibility.

multi-agent system

A workflow where multiple specialized agents coordinate, pass outputs, and contribute to a larger task.

That makes MAS development less like writing one prompt and more like designing a stochastic, interconnected software pipeline.

finding 1

Developers describe a two-stage workflow

A recurring pattern is to first tune or validate individual agents in isolation, then move to end-to-end workflow testing where coordination failures begin to surface.

stage 1: local agent testing

Prompt a single agent, inspect examples, patch instructions, and push on obvious failure modes.

fast feedback role-level iteration
→

stage 2: workflow validation

Run the full chain and watch whether outputs remain stable once agents depend on one another.

slower feedback cross-agent failures

important implication: the critical transition is not “does one prompt work?” but “does the whole chain stay reliable when every downstream step depends on upstream output quality?”

finding 2

Much of the time goes to slow trial-and-error

Because the sample leans industry-facing, participants repeatedly emphasized consistency, stability, and edge-case handling rather than novelty alone.

edge cases keep appearing late

Developers rerun agents or entire workflows many times to expose rare but costly failures.

consistency matters as much as quality

Acceptable outputs are not enough if the workflow is unstable across repeated runs or nearby inputs.

prompts grow with project complexity

Prompts accumulate instructions, company rules, edge-case patches, and collaborator edits over time.

starting from zero is hard

Some developers feel prompt writing itself is not their expertise, so the first version already feels underspecified.

finding 3

MAS debugging differs from traditional programming

Since these systems are stochastic and chained, failure is often diffuse: one weak output can silently degrade the rest of the workflow, and the source is rarely obvious.

Compared with traditional software, developers do not get a stable fault taxonomy or a clean stack trace. They often end up reading long logs and manually tracing where the breakdown first entered the chain.

bridge

From interviews to artifact analysis

If developers struggle with complex, growing prompts, then it becomes useful to look directly at prompts from existing projects and ask whether there are recurring roles and recurring internal structures.

This is the motivation for the component analysis stage.

If common role families exist...

we could build better kickstart templates instead of forcing everyone to write a long system prompt from scratch.

If common prompt sections exist...

we could frame prompts as components or sections, giving developers a clearer mental model than one large block of text.

working idea: recurring structure could support more organized authoring, comparison, and debugging interfaces for MAS development.

scope + visibility

LLM-based MAS with at least three separate agents or clearly differentiated roles.

Repository must be open-source and prompt text must be explicitly visible.

system definition

Built on a common MAS framework or a custom orchestration with clearly described multi-agent roles.

This keeps the dataset focused on actual agent workflows rather than single-prompt apps.

activity + traction

Recent commit within roughly the last 12-18 months.

More than 100 stars or more than 10 forks as a rough visibility threshold.

analysis pipeline

How the prompt analysis works

The pipeline combines component coding, embeddings, clustering, and LLM-assisted summarization to surface recurring prompt roles and their characteristic sections.

1

component coding

Use a codebook and LLM support to mark whether prompts contain identifiable components or sections.

2

embedding + clustering

Embed each prompt and group similar prompts to identify recurring role families.

3

cluster labeling

Generate a short human-readable summary for each cluster.

4

distribution comparison

Compare which components are characteristic of each cluster or role family.

takeaway

The larger goal is practical support for MAS builders

empirical grounding

Use interviews to characterize how MAS development actually unfolds and where the friction concentrates.

role-aware templates

If role families recur, provide stronger starting points than blank prompt editors.

componentized prompt views

If prompt structure recurs, support editing and comparison by sections instead of one long monolith.

better debugging surfaces

Design interfaces around cross-agent dependencies, intermediate outputs, and failure tracing.

current working hypothesis

MAS development is difficult not only because prompts are hard to write, but because developers are coordinating stochastic components without the debugging affordances they are used to in traditional software.
